## 1. 引言与适用范围

### 1.1 文档目的

本文档旨在为投资模拟游戏系统（项目代号：2026-05-23-task-2）建立一套完整、规范、可执行的漏洞报告策略与处理流程。本策略面向所有与本项目交互的外部安全研究者、社区贡献者以及内部开发团队成员，提供从漏洞发现到修复关闭的全生命周期管理标准。

漏洞报告策略的核心目标包括：

1. **统一入口**：为安全研究者、社区开发者、内部测试人员提供一个标准化的漏洞提交渠道，避免碎片化沟通导致的信息遗失。
2. **规范流程**：定义从漏洞提交（Submitted）到最终关闭（Closed）的完整状态机，确保每个漏洞都有迹可循、有据可查。
3. **质量保障**：通过标准化的报告模板和分类体系，减少沟通成本，加速漏洞的定位与修复。
4. **社区激励**：建立公平透明的赏金与致谢机制，鼓励安全社区积极参与项目的安全建设。
5. **持续改进**：将漏洞发现作为安全开发生命周期的反馈输入，推动代码质量和安全架构的持续提升。

### 1.2 适用范围

本策略适用于投资模拟游戏系统的以下全部模块：

#### 核心中枢模块

- **invest-sim.html**（495KB / 8715行）：投资模拟主控平台。该文件是整个系统的唯一数据中枢，管理 9 类资产（现金、科技股、能源股、医疗股、矿业股、比特币、国债、P2P 债权、黄金、期货）的动态价格模拟、交易日志、止损止盈、净值曲线与雷达图渲染。通过 `localStorage` 中 `investSimState_v13` 键维护全局状态，通过 `invest_sim_master` 键产出母锁令牌。

#### 金融子模块群（11 家银行 + 9 家 VPN + 2 家券商 + 3 家 P2P 平台 + 3 家保险公司）

- **银行模块（bank-*.html × 11）**：由 `gen_banks.py` 数据驱动生成的银行信息展示页面，覆盖中国内地（工商银行、建设银行、中国银行、农业银行）、中国香港（汇丰、恒生、中银香港、渣打）、英国（英格兰银行、汇丰英国）三大金融区域，共计 29 张银行卡与关联持卡人数据。
- **VPN 模块（vpn-*.html × 9）**：由 `gen_vpn.py`（91KB / 2954行）生成的银行 VPN 内网系统门户，提供登录验证后的六大功能标签页（账户总览、交易流水、余额查询、转账、安全设置、日志）。采用 CRT Matrix 绿色终端主题。
- **券商模块（broker-citics.html / broker-goldman.html）**：由 `gen_brokers.py` 生成的中信证券与高盛证券交易平台。实现三层"锁链"令牌验证机制（母锁 → 子锁 → 数据锁），支持 4 类股票（科技、能源、医疗、矿业）的持仓查询、委托下单与成交查询。
- **P2P 模块（p2p-lufax.html / p2p-ppdai.html / p2p-renrendai.html）**：由 `gen_p2p.py` 生成的陆金所、拍拍贷、人人贷三大借贷平台。各个平台有不同的风险等级和违约概率（陆金所 3%、拍拍贷 8%、人人贷 5%），含 10 种贷款项目模板。
- **保险模块（insurance-aia.html / insurance-picc.html / insurance-pingan.html）**：由 `gen_insurance.py` 生成的友邦保险、中国人保、平安保险三大保单管理平台。每家包含 3 种保险产品（健康险、财产险、意外险）和 10 种理赔事件模板。

#### 黑客子模块群

- **world-server.html**（149KB / 3108行）：万维网攻防模拟平台。模拟全球互联网 10 个网络集群（电商云、政府内网、金融科技、社交媒体等）的服务器渗透攻击。集成 0day 漏洞发现系统（通过 `nmap` 扫描有概率发现未公开 CVE），攻击成功后生成 `dataFragments` 数据碎片并通过 `syncFragmentsToInvestSim()` 写入投资模拟器状态。
- **bank-attack.html**（61KB）：银行渗透攻击专用平台。针对 11 家银行的 VPN 渗透攻击，追踪得分（score）、信誉（rep）、追踪热度（trace）三项黑客指标。
- **darknet-market.html**（40KB）：Tor 风格暗网交易平台。支持出售银行数据残骸、VPN 凭据和世界服务器数据碎片，集成 FBI 追踪热度机制（`fbiHeat` 0~100，≥80 触发突袭检查）和 0day 漏洞交易。
- **hack-training.html**（60KB）：CRT 终端风格的渗透学习平台，含 PII 数据库、SQL 注入教程和密码破解训练模块。
- **hack-terminal.html**（28KB）：通用网络渗透终端模拟器，提供 `scan` / `exploit` / `bruteforce` 等交互命令。

#### 辅助工具模块

- **atm.html**（40KB）：CRT 终端风格 ATM 取款机模拟器，支持插卡、查询、取款、转账和 PDF 凭条导出。
- **remittance.html**（31KB）：跨境汇款工具，展示内地-香港-英国三地银行账户之间的资金转移。

#### 代码生成器与构建工具

- **Python 生成器**（6 个）：`gen_banks.py`、`gen_vpn.py`、`gen_brokers.py`、`gen_p2p.py`、`gen_insurance.py`、`gen_accounts_excel.py`。这些脚本通过数据驱动方式批量产出结构一致的 HTML 子页面，是本项目的核心构建环节。
- **JavaScript 构建脚本**（10 个）：用于功能注入、语法修复和模块集成的 Node.js 脚本。
- **独立 JS 模块**：`darknet-ws-trade.js`（148 行），提供 `escapeHtml()` 函数和数据碎片交易逻辑。

### 1.3 漏洞报告策略的定位

本策略作为项目的**统一安全入口**，同时服务于以下三类人群：

| 人群 | 角色 | 与项目的关系 |
|------|------|------------|
| **内部开发者** | 代码编写者与维护者 | 在开发过程中发现自身或他人代码的安全缺陷，通过本流程提交并追踪修复 |
| **安全研究者** | 外部白帽黑客与安全专家 | 通过安全测试、代码审计或渗透测试发现漏洞并报告，预期获得赏金或致谢 |
| **社区贡献者** | 开源社区参与者 | 在使用或审阅代码过程中发现安全隐患，通过本流程协助项目改进 |

这三种人群共享同一套报告模板和提交流程，但在赏金资格判定上有所区别（详见第六章）。

### 1.4 与项目整体安全架构的关系

本项目的安全架构已具备以下设计基础，漏洞报告策略是这些已有安全机制的延伸与补充：

#### 1.4.1 三层锁链令牌机制

项目实现了从核心中枢到金融子模块的三层递进式鉴权体系：

- **母锁（Master Lock）**：由 `invest-sim.html` 的 `refreshMasterToken()` 方法生成，存储于 `localStorage` 的 `invest_sim_master` 键中。令牌结构为 `{ts: 时间戳, seal: 随机签名, status: 'active'}`，有效期 1 小时（`MASTER_TOKEN_TTL = 3600000`）。每次 `saveState()` 调用时自动刷新。
- **子锁（Sub Lock）**：券商页面（`broker-*.html`）在母锁验证通过后本地生成的会话级令牌，绑定当前浏览器标签页的访问生命周期。
- **数据锁（Data Lock）**：对 `localStorage` 中持仓数据访问的二次加密校验，防止通过浏览器开发者工具直接篡改状态数据绕过业务逻辑。

任何子页面在加载时首先读取 `invest_sim_master`，依次校验 `ts` 时效性、`seal` 签名完整性和 `status` 激活状态。验证未通过则拒绝展示功能界面，返回错误提示。这一机制在 `gen_brokers.py` 生成的券商页面中实现为 `validateMasterToken()` 和 `updateLockChain()` 两个函数，提供了可视化的锁链状态指示器。

漏洞报告策略在这一架构中的角色是：当安全研究者发现令牌验证存在绕过可能、token 结构存在字段名不一致（如已修复的 Bug #3，`token.time` vs `token.ts`）、或三层锁链中某一层的校验逻辑存在缺陷时，提供标准化的报告渠道。

#### 1.4.2 0day 漏洞系统

在 `world-server.html`（万维网攻防模拟平台）中实现了完整的 0day 漏洞生命周期管理：

- **发现**：玩家对服务器集群执行 `nmap` 扫描时，有基础 50% 概率发现未公开 CVE 漏洞。已发现 0day 数量越多，发现新 0day 概率越低（每个递减 15%）。
- **存储**：发现的 0day 存入 `data.zeroDaysDiscovered[]` 数组，每个漏洞含 `{id, cve, name, grade, decayRounds, used}` 元数据，并同步写入 `investSimState_v13`。
- **使用**：攻击服务器时自动选择最优 0day 提升成功率，使用后有 50% 概率暴露失效。通过 `consumeBestZeroDay()` 函数管理消耗逻辑。
- **过期**：0day 有效期到期后自动标记失效（模拟厂商修补），由 `processZeroDayDecay()` 函数处理。

这一系统为漏洞报告策略提供了天然的参考框架——安全研究者报告的 0day 漏洞在游戏世界中对应高价值交易品（暗网市场中 0day 售价 ₿5.0，风险指数 15），反映了项目对漏洞价值的认知。

#### 1.4.3 FBI 追踪热度系统

`fbiHeat` 是贯穿金融与黑客双主题的核心风险指标，模拟真实世界中执法机构对网络犯罪行为的追踪压力：

- **初始值**：0（无追踪）
- **增长机制**：每次在暗网市场出售数据碎片、0day 漏洞或银行数据残骸时，`fbiHeat` 增加相应风险值（碎片 +5、0day +15、银行残骸 +10）
- **阈值触发**：`fbiHeat >= 50` 时 UI 显示红色警告；`fbiHeat >= 80` 时每次交易后有 15% 概率触发 FBI 突袭
- **突袭结果**：触发 `triggerFBIRaid()` 函数，随机扣除资产、降低信誉、清空部分数据
- **清零机制**：`fbiHeat >= 100` 时自动强制清零（表示已被控制），或通过特定反侦察操作主动降低

这一系统为漏洞报告策略提供了威胁建模的参考——就像 FBI 热度模拟了外部威胁的逐步升级，漏洞严重度分级也应反映漏洞被利用后的潜在影响逐步升级。

#### 1.4.4 XSS 防护实践

项目中已在多处实现了基础的跨站脚本攻击（XSS）防护：

- **`escapeHtml()` 函数**：在全部 11 家银行页面（`bank-*.html`）、2 家券商页面（`broker-*.html`）和暗网碎片交易模块（`darknet-ws-trade.js`）中均有定义。实现方式为 `var d = document.createElement("div"); d.textContent = str || ""; return d.innerHTML;`，利用浏览器 DOM API 的内建转义能力，比正则替换更可靠。
- **输出编码应用场景**：银行页面的交易流水渲染（`escapeHtml(r.name)`、`escapeHtml(r.bankType)`、`escapeHtml(r.source)`）、券商页面的持仓列表渲染、暗网碎片交易的数据展示。
- **`textContent` 优先**：在最终版本的碎片交易模块（`darknet-ws-trade.js`）中，标题和按钮文本使用 `textContent` 而非 `innerHTML`，从根本上避免了 XSS 注入路径。

这些实践表明项目已具备基本的安全编码意识，漏洞报告策略将在此基础上建立更系统的安全测试标准。

## 2. 漏洞分类体系

### 2.1 按严重程度分级

漏洞严重度分级采用五级制，综合考量**可利用性**（Exploitability）、**影响范围**（Scope）、**数据敏感度**（Data Sensitivity）和**攻击复杂度**（Attack Complexity）四个维度。每一级别的判定标准如下：

#### 2.1.1 Critical（严重）

**定义**：可直接导致系统核心功能完全瘫痪、敏感数据大批量泄露、或使攻击者获得最高权限的漏洞。无需用户交互或仅需极少交互即可触发。

**判定标准**（满足任意一项即为 Critical）：

| 维度 | 标准 |
|------|------|
| 可利用性 | 无需认证、无需用户交互、无需特定条件即可远程触发 |
| 影响范围 | 影响全部模块的所有用户，或导致 `investSimState_v13` 全局状态被完全控制 |
| 数据敏感度 | 可读取/篡改所有 `localStorage` 数据，或绕过三层锁链令牌验证获取所有金融子模块的完整控制权 |
| 攻击复杂度 | 极易利用，不依赖复杂环境或时间窗口 |

**具体场景示例（结合本项目代码）**：

1. **母锁令牌伪造漏洞**：如果攻击者能够在不访问 `invest-sim.html` 的情况下，通过分析 `refreshMasterToken()` 的随机数生成逻辑（`Math.random().toString(36).slice(2) + Date.now().toString(36)`），预测或重放 `seal` 签名，从而构造合法令牌绕过三层锁链验证，获取所有券商、P2P、保险页面的完整访问权限——这将被定为 Critical。

2. **`localStorage` 跨站脚本写入漏洞**：如果 `darknet-market.html` 或 `world-server.html` 的任何用户输入点在未经过 `escapeHtml()` 处理的情况下被写入 `localStorage` 中的 `investSimState_v13`，而 `invest-sim.html` 读取后直接通过 `innerHTML` 渲染到页面——这将形成存储型 XSS，影响所有打开投资中枢的标签页。因涉及全局状态污染，定为 Critical。

3. **0day 漏洞发现概率溢出**：在 `world-server.html` 的 `discoverZeroDay()` 函数中，如果概率计算逻辑存在整数溢出或边界条件错误，导致攻击者能以 100% 概率无限量发现 0day，进而在暗网市场无限刷取 BTC（每个 0day 售价 ₿5.0）——这将破坏整个数据碎片经济系统，定为 Critical。

4. **三层锁链的全局绕过**：如果券商页面的 `validateMasterToken()` 函数存在逻辑短路（如 `if (!raw) { ... }` 后未正确 `return`，继续执行后续代码），导致即使母锁令牌不存在也能通过验证——这一定为 Critical。

#### 2.1.2 High（高）

**定义**：可导致单模块功能完全瘫痪、敏感数据选择性泄露、或使攻击者获得部分权限提升的漏洞。可能需要一定的用户交互或特定条件。

**判定标准**（满足任意一项即为 High）：

| 维度 | 标准 |
|------|------|
| 可利用性 | 需要有限的认证或特定条件（如已知某个子页面的 URL），但无需复杂的用户交互 |
| 影响范围 | 影响单一模块群（如所有 P2P 页面或所有券商页面），但不波及其他模块 |
| 数据敏感度 | 可读取特定类别的 `localStorage` 数据（如仅能读取 `p2pInvestments` 而不能读取 `assets`） |
| 攻击复杂度 | 中等，需要一定的技术手段但非极端困难 |

**具体场景示例**：

1. **单个生成器脚本的模板注入**：如果 `gen_p2p.py` 的 HTML 模板字符串中存在未转义的用户可控变量（如直接将贷款项目标题拼接进 `innerHTML`），导致生成的 3 家 P2P 页面存在 DOM 型 XSS——影响范围限于 P2P 模块，定为 High。

2. **P2P 违约概率篡改**：如果攻击者通过浏览器控制台直接修改 `localStorage` 中 `investSimState_v13` 的 P2P 平台配置，将陆金所的违约率从 3% 改为 0%，然后 P2P 子页面未进行服务端（代码层面）的违约率范围校验——这允许攻击者在 P2P 模块获得不当利益，定为 High。

3. **子锁/数据锁的单点绕过**：如果三层锁链中"子锁"或"数据锁"的验证逻辑存在缺陷，但母锁仍然有效——攻击者未经授权可查看券商持仓数据但无法修改——定为 High。

4. **暗网市场 FBI 突袭绕过**：如果 `sellWorldFragment()` 函数中 FBI 热度累积逻辑存在错误（如 `fbiHeat` 增加值被错误计算为 0），导致攻击者可以无限量出售数据碎片而永远不触发 FBI 突袭——定为 High。

#### 2.1.3 Medium（中）

**定义**：可导致非核心功能异常、有限的敏感信息泄露、或需要复杂的攻击链才能产生实际危害的漏洞。

**判定标准**：

| 维度 | 标准 |
|------|------|
| 可利用性 | 需要多步用户交互或特定的攻击链组合才能触发 |
| 影响范围 | 影响单个页面或非关键功能，不影响核心业务流程 |
| 数据敏感度 | 信息泄露限于低敏感度数据（如公开的 UI 状态信息） |
| 攻击复杂度 | 较高，需要利用链或多条件组合 |

**具体场景示例**：

1. **CSS 注入导致 UI 欺骗**：如果子页面的某个 CSS 类名由用户输入控制且未做白名单校验，攻击者可通过注入样式伪造 UI 元素（如将"卖出"按钮伪装成"买入"）——因需要用户实际点击才能造成损失，定为 Medium。

2. **交易日志信息泄露**：如果 `invest-sim.html` 的交易日志渲染时，某个日志字段（如内部调试信息）意外暴露在浏览器控制台或页面注释中，但该信息不包含敏感凭据——定为 Medium。

3. **竞态条件导致的数据不一致**：如果攻击者同时打开多个标签页，利用 `localStorage` 无锁特性，在券商下单和核心中枢处理之间制造竞态窗口，导致持仓数据出现短暂不一致——因需要精确的时间窗口且影响有限，定为 Medium。

4. **生成的 HTML 结构错误**：如已修复的 Bug #5（sell 按钮前多余的 `<` 前缀）或 Bug #6/#7（class 属性引号错位），如果这些问题可被恶意利用（如通过属性注入劫持事件处理）——但实际攻击路径狭窄，定为 Medium。

#### 2.1.4 Low（低）

**定义**：不影响系统功能和安全性的轻微缺陷，理论上的攻击面但实际利用价值极低。

**判定标准**：

| 维度 | 标准 |
|------|------|
| 可利用性 | 几乎无法在实际环境中利用，或需要完全不现实的假设条件 |
| 影响范围 | 极小，仅限于单个非敏感 UI 元素或日志信息 |
| 数据敏感度 | 无敏感数据泄露 |
| 攻击复杂度 | 极高，需要攻击者具备物理访问权限或其他极端条件 |

**具体场景示例**：

1. **`localStorage` 容量耗尽导致的功能降级**：如果攻击者通过大量写入垃圾数据消耗浏览器 `localStorage` 配额（通常 5-10MB），导致 `saveState()` 写入失败——但这一操作无法获取任何信息或权限，且容易恢复——定为 Low。

2. **已失效的调试代码残留**：如果某个子页面的 HTML 注释中包含已过期的开发调试信息（如代码片段注释），不包含凭据或敏感路径——定为 Low。

3. **不安全的随机数使用**：`refreshMasterToken()` 使用 `Math.random()` 生成 `seal`。虽然 `Math.random()` 不是密码学安全的随机数生成器，但在纯前端 localStorage 共享状态下，该令牌的作用是跨页面协调而非安全防护（真正的安全边界不存在于同源的 localStorage 中），因此实际风险较低。如果将此场景迁移到服务端环境则升级为 High。

#### 2.1.5 Informational（信息性）

**定义**：不构成漏洞但值得关注的安全相关建议，包括代码质量改进、最佳实践推荐、架构层面的安全加固建议等。

**具体场景示例**：

1. **建议引入 Content Security Policy（CSP）**：当前所有 HTML 页面未设置 CSP 头或 `<meta>` 标签，建议添加以防御 XSS 和数据注入攻击。
2. **建议对 `localStorage` 数据做 JSON Schema 校验**：当前任何子页面可直接写入 `investSimState_v13` 的任意字段，无结构校验机制，建议在 `saveState()` 中增加 Schema 验证。
3. **建议将 `escapeHtml()` 抽取为公共模块**：当前该函数在 22+ 个文件中各自独立定义，代码重复严重，维护时容易遗漏更新。
4. **建议对生成器脚本进行安全审计**：6 个 Python 生成器直接操作文件系统并产出 HTML，若模板变量未正确转义可能导致生成包含恶意代码的页面。

### 2.2 按漏洞类型分类

#### 2.2.1 注入类漏洞

**2.2.1.1 跨站脚本注入（XSS）**

本项目的 XSS 攻击面主要集中在以下几个环节：

- **存储型 XSS**：由于所有模块共享 `localStorage` 中的数据（`investSimState_v13`、`worldServerState` 等），任何一个写入点如果未经过 `escapeHtml()` 或 `textContent` 安全处理，可能形成存储型 XSS。例如：
  - `world-server.html` 通过 `syncFragmentsToInvestSim()` 写入的 `worldFragments[]` 数组中的 `name`、`fromCluster`、`fromServer` 字段
  - P2P 子页面写入 `p2pInvestments` 的贷款项目标题
  - 保险子页面写入 `insurancePolicies` 的理赔事件描述

- **DOM 型 XSS**：如果页面通过 URL 参数（`location.hash`、`location.search`）或 `postMessage` 等渠道接收数据后未经转义直接写入 DOM——当前项目主要通过 `localStorage` 而非 URL 通信，但 `remittance.html` 可能通过 `window.open()` 传递参数，需关注。

- **属性注入**：已修复的 Bug #6/#7（class 属性引号错位）如果在其他生成器中类似存在，可能导致攻击者通过精心构造的数据值注入 HTML 属性，劫持事件处理器或样式。

**2.2.1.2 SQL 注入（教学场景）**

`hack-training.html` 中包含了 SQL 注入的教学模块，模拟了拼接 SQL 查询的场景。虽然这些 SQL 语句仅在客户端解释执行（无真实数据库），但教学模块中展示的漏洞模式可能被误用于其他场景，需要明确标注"仅供教学使用"。

**2.2.1.3 DOM 注入**

项目中大量使用 `innerHTML` 进行动态渲染（`invest-sim.html` 的 `renderAssets()`、`renderMarginTab()` 等），如果这些渲染函数的数据源来自用户可控的 `localStorage` 值且未经过 `escapeHtml()` 处理，将形成 DOM 注入点。

#### 2.2.2 信息泄露类漏洞

**2.2.2.1 敏感信息明文存储**

本项目的核心架构决定了所有数据（包括模拟的银行账户信息、VPN 凭据、0day 漏洞数据）均以明文 JSON 形式存储在浏览器的 `localStorage` 中。这在纯前端游戏中属于可接受的架构选择（无可信服务端），但需注意：

- `gen_banks.py` 中定义的 VPN 凭据（如 `vpnCredentials`）被硬编码在 Python 脚本中，并最终生成到 HTML 文件。如果这些凭据曾经用于任何真实系统的测试，将构成严重的信息泄露。
- `gen_accounts_excel.py` 生成的 `账户清单.xlsx` 包含 29 张银行卡号和 6 个证券账户信息，虽然是模拟数据，但如果数据格式与真实卡号相似，可能被误用。

**2.2.2.2 调试信息暴露**

- `invest-sim.html` 的 `addLog()` 函数可能将内部状态信息输出到交易日志 UI 中。如果日志渲染时未过滤敏感字段（如临时调试的 token 值），将导致信息泄露。

**2.2.2.3 浏览器开发者工具暴露**

由于所有逻辑均在客户端执行，攻击者可以通过浏览器开发者工具直接查看和修改所有 JavaScript 变量、函数逻辑和 localStorage 数据。这在纯前端架构中无法完全避免，但应通过代码混淆（如果必要）和明确的"客户端可见"声明来管理预期。

#### 2.2.3 逻辑缺陷类漏洞

**2.2.3.1 鉴权绕过**

三层锁链令牌机制是本项目的核心安全边界。以下场景均属于鉴权绕过：

- **令牌时效性绕过**：`validateMasterToken()` 在校验 `age > 3600000` 时，如果系统时间被篡改或 `Date.now()` 的精度（毫秒）被利用，可能绕过有效期限制。
- **令牌字段名不匹配**：已修复的 Bug #3 是最典型的鉴权绕过——由于 `gen_p2p.py` 和 `gen_insurance.py` 读取 `token.time` 而 `refreshMasterToken()` 写入的字段名为 `token.ts`，读取结果为 `undefined`，经过 `Date.now() - undefined` 计算得到 `NaN`，导致任何时效校验逻辑失效，所有子页面永远无法通过验证。
- **令牌覆盖攻击**：已修复的 Bug #4——`processInsurance()` 在保险回调时自行构造了另一套 token 格式 `{value, issuedAt, expiresAt}`，覆盖了 `refreshMasterToken()` 写入的 `{ts, seal, status}` 格式，导致后续券商页面读取到的 token 格式不被识别。

**2.2.3.2 竞态条件**

由于多标签页通过 `localStorage` 共享状态且无锁机制，以下竞态条件可能存在：

- **下单与价格更新竞态**：券商页面下单后写入 `investSimState_v13.assets`，同时 `invest-sim.html` 的价格模拟引擎（`simulateRound()`）也在更新同一 `assets` 数组中的 `price` 字段。如果在写入和读取之间存在时间窗口，可能导致订单以过期价格成交。
- **碎片出售双重花费**：`darknet-market.html` 中 `sellWorldFragment()` 通过 `frags.splice(idx, 1)` 移除已售碎片。如果攻击者快速连续点击（或通过脚本快速调用），可能在第一次 `splice` 尚未写回 `localStorage` 时第二次调用也通过了 `idx >= 0` 检查。

**2.2.3.3 状态不一致**

- **数据碎片同步断裂**：`world-server.html` 攻破服务器后通过 `syncFragmentsToInvestSim()` 将碎片写入 `investSimState_v13.worldFragments[]`，但 `worldServerState` 中也有独立的 `player.dataFragments` 数组。如果两个数组不同步（例如一个操作只更新了其中之一），将导致暗网市场显示的碎片列表与实际可出售的碎片列表不一致。

#### 2.2.4 配置安全类漏洞

**2.2.4.1 权限缺失**

- **直接 localStorage 写入**：任何同源页面都可以直接通过 `localStorage.setItem('investSimState_v13', ...)` 写入任意数据，无写权限控制。虽然在同一浏览器同源策略下这是设计如此，但项目没有实现写入前的数据签名校验或版本检查。

**2.2.4.2 默认凭据**

- 生成器脚本中定义的银行账户、VPN 凭据、P2P 平台数据都是模拟数据。但如果未来这些生成器被复用于其他需要真实安全防护的项目，这些"默认凭据"将构成严重风险。需在生成器脚本头部添加明确的免责声明。

**2.2.4.3 不安全的通信**

- 所有页面间通信通过 `localStorage` 明文进行。如果浏览器运行环境中安装了恶意扩展，可以读取所有 `localStorage` 数据。这是浏览器扩展安全模型的内在限制，但应在安全文档中明确说明。

  摘要：
  投资中枢通过 refreshMasterToken() 生成母锁令牌，写入

  生成器在验证令牌时读取的字段名为 time（错误字段名），导致
  token.ts 的值为 undefined，时效校验 Date.now() - undefined

  - 以及对应的备份/副本文件

  字段名 token.ts，因此券商页面不受影响。这种不一致表明
  该问题是由于多个生成器之间的代码复制粘贴且缺乏统一
  的验证函数库所致。

  预期行为：
  打开 p2p-lufax.html 后，页面应读取 invest_sim_master

  实际行为：

  令牌验证判定为"已过期"，页面拒绝展示任何功能，显示

### 3.3 PGP/GPG 加密通信指引

对于包含敏感细节的漏洞报告（如涉及可实际利用的攻击代码、尚未修复的严重漏洞细节等），报告人可以选择使用 PGP/GPG 加密方式与项目安全团队通信。

**项目安全团队公钥**：

```
-----BEGIN PGP PUBLIC KEY BLOCK-----

[此处为项目安全团队的 PGP 公钥，在正式发布本文档时需替换为实际公钥]

-----END PGP PUBLIC KEY BLOCK-----
```

**加密通信流程**：

1. 报告人使用上述公钥加密漏洞报告的完整内容。
2. 将加密后的文本通过电子邮件发送至项目安全邮箱（见第四章）。
3. 邮件主题格式：`[ENCRYPTED] 漏洞报告 — [严重度] — [模块名]`
4. 安全团队使用私钥解密后，通过报告人提供的联系渠道（同样加密）确认收悉。
5. 后续关于该漏洞的所有通信均使用 PGP/GPG 加密。

**注意事项**：

- 请勿将私钥或解密密码附在报告邮件中。
- 如果报告人没有 PGP/GPG 使用经验，可以跳过加密直接发送明文报告（项目安全邮箱仅供安全报告使用，不作为公开列表）。
- 敏感漏洞（Critical / High）建议始终使用加密通信，即使这样会稍微增加初次沟通的时间成本。

## 4. 报告提交流程

### 4.1 提交渠道

本项目提供以下三种漏洞提交渠道，按推荐优先级排列：

| 优先级 | 渠道 | 适用场景 | 响应速度 |
|--------|------|---------|---------|
| 1（推荐） | GitHub Security Advisory | 所有漏洞，特别是希望追踪处理进度的公开漏洞 | 48小时内 |
| 2 | 项目安全邮箱 `security@invest-sim.dev` | 敏感漏洞、需要加密通信的漏洞 | 72小时内 |
| 3 | GitHub Issue 模板（`Bug Report — Security`） | 低严重度漏洞（Low / Informational）、信息性建议 | 5个工作日内 |

**渠道 1：GitHub Security Advisory**

GitHub Security Advisory 是推荐的首选渠道，因为它提供了：
- 私密报告能力（在修复发布前仅安全团队可见）
- 自动化的 CVE 编号申请流程
- 完整的讨论历史记录
- 与项目代码仓库的直接关联

提交步骤：
1. 访问项目 GitHub 仓库的 Security 标签页
2. 点击 "Report a vulnerability"
3. 按照第三章的标准模板填写报告内容
4. 选择严重度等级
5. 提交后，安全团队将在 48 小时内确认收悉

**渠道 2：安全邮箱**

适用于希望使用 PGP/GPG 加密通信、或者没有 GitHub 账号的报告人。发送邮件时请遵循以下格式：

```
收件人：security@invest-sim.dev
主题：[VULN] invest-sim — [严重度] — [模块] — [简述]

邮件正文：
按第三章标准模板填写，如果是加密邮件请注明 [ENCRYPTED]
```

**渠道 3：GitHub Issue**

适用于低严重度漏洞和一般性安全建议。请在提交时选择 "Bug Report — Security" 模板（如果仓库已配置），或在标题前添加 `[SECURITY]` 前缀。

### 4.2 完整生命周期状态机

漏洞从提交到关闭经过以下 6 个状态，每个状态有明确的判定标准和流转条件：

```
  ┌──────────┐
  │Submitted │  报告人提交漏洞报告
  └────┬─────┘
       │ 安全团队在 SLA 时间内确认收悉
       ▼
  ┌──────────┐
  │  Triage  │  安全团队审核报告，评估严重度和真实性
  └────┬─────┘
       │
       ├─── 拒绝（不相关/重复/非安全问题）────→ Closed (Won't Fix)
       │
       ├─── 降级为 Informational ──────────→ Closed (Informational)
       │
       └─── 确认为有效漏洞
            │
            ▼
       ┌──────────┐
       │Confirmed │  漏洞被确认，分配开发资源，排入修复计划
       └────┬─────┘
            │
            ▼
       ┌──────────┐
       │In Progress│  开发者正在修复漏洞
       └────┬─────┘
            │
            ├─── Critical 走热修复快速通道
            │
            └─── 修复完成，进入测试验证
                 │
                 ▼
            ┌──────────┐
            │ Resolved │  修复已合并，等待最终验证和公告
            └────┬─────┘
                 │
                 ▼
            ┌──────────┐
            │  Closed  │  漏洞生命周期结束
            └──────────┘
```

#### 4.2.1 Submitted（已提交）

**进入条件**：报告人通过任一渠道提交了完整的漏洞报告。

**安全团队操作**：
- 自动回复确认收悉（通过 GitHub Advisory 或邮件自动回复）
- 记录提交时间戳，开始 SLA 计时

**流转条件** → **Triage**：安全团队在 SLA 时间内开始审核。

#### 4.2.2 Triage（审核中）

**进入条件**：安全团队成员认领了该漏洞报告，开始审核。

**安全团队操作**：
1. **验证真实性**：按照报告中的复现步骤尝试重现漏洞。
   - 如果无法复现，联系报告人补充信息（状态暂停但 SLA 不重置）。
   - 如果确认无法复现且无补充信息，标记为 `Closed (Cannot Reproduce)`。
2. **评估严重度**：对照第二章的分类标准，确认或调整报告的严重度评级。
   - 如果安全团队的评定与报告人不同，需在评论中说明理由。
3. **检查重复**：搜索已有漏洞记录，确认是否与之前报告的漏洞重复。
   - 如果重复，标记为 `Closed (Duplicate)` 并链接到原始漏洞。
4. **确定范围**：确认漏洞是否属于本策略适用范围（见第一章）。
   - 如果不在范围内（如第三方依赖库的已知漏洞且项目未使用受影响版本），标记为 `Closed (Out of Scope)`。

**流转条件**：
- → **Confirmed**：漏洞确认有效且不在上述拒绝条件中。
- → **Closed**：漏洞被拒绝。

#### 4.2.3 Confirmed（已确认）

**进入条件**：漏洞已被确认为真实有效的安全问题。

**安全团队操作**：
1. 为漏洞分配唯一编号（格式：`ISV-YYYY-NNN`，如 `ISV-2026-001`）。
2. 确定修复优先级：
   - **P0（紧急）**：Critical 漏洞，需要在 24 小时内发布热修复。
   - **P1（高优先）**：High 漏洞，纳入当前或下一个 Sprint。
   - **P2（正常）**：Medium 漏洞，纳入未来两个 Sprint。
   - **P3（低优先）**：Low / Informational，放入 Backlog。
3. 指定负责修复的开发者。
4. 如果是通过 GitHub Advisory 提交的，此时可申请 CVE 编号（如适用）。

**流转条件** → **In Progress**：开发者开始编写修复代码。

#### 4.2.4 In Progress（修复中）

**进入条件**：开发者开始编写漏洞修复代码。

**开发者操作**：
1. 创建修复分支（如 `fix/ISV-2026-001-token-field-mismatch`）。
2. 编写修复代码，含必要的单元测试或回归测试。
3. 提交 Pull Request，在 PR 描述中引用漏洞编号。
4. 代码审查（Code Review）通过后合并到主分支。

**安全团队操作**：
- 对修复进行安全性验证——确认修复不会引入新的安全问题。
- 对于生成器脚本的修复，需同时更新生成器脚本并重新生成所有受影响的 HTML 子页面。
- 确保类似问题模式不会在其他生成器中重复出现（如 Bug #3 的修复需同时检查全部 6 个生成器的 token 读取逻辑）。

**流转条件** → **Resolved**：修复代码已合并，通过安全验证。

#### 4.2.5 Resolved（已解决）

**进入条件**：修复已部署（对于纯前端项目，这意味着修复后的 HTML 文件已更新）。

**安全团队操作**：
1. 通知报告人漏洞已修复，邀请其验证。
2. 准备安全公告草稿（如适用）。
3. 根据严重度确定披露时间线（见第五章 SLA 表）。
4. 如果申请了 CVE 编号，等待 CVE 发布后更新公告中的编号。

**流转条件** → **Closed**：满足以下任一条件：
- 报告人确认修复有效。
- 安全公告已发布（如果适用）。
- 修复部署后经过充分的公示期（Critical: 30天, High: 14天, Medium/Low: 7天）。

#### 4.2.6 Closed（已关闭）

**进入条件**：漏洞生命周期完成。

**关闭原因分类**：

| 关闭原因 | 含义 |
|---------|------|
| `Fixed` | 漏洞已修复并验证 |
| `Won't Fix` | 不修复（不在范围内 / 不可利用 / 风险可接受） |
| `Duplicate` | 与已有漏洞重复 |
| `Cannot Reproduce` | 无法复现 |
| `Informational` | 信息性建议，非安全漏洞 |
| `Out of Scope` | 超出适用范围 |

### 4.3 紧急热修复流程（Critical 漏洞的 Expedited 通道）

对于被评定为 Critical 级别的漏洞，走以下快速通道：

```
Critical 漏洞提交
       │
       ▼
  [1小时内] 安全团队负责人确认并通知全体核心开发者
       │
       ▼
  [4小时内] 完成漏洞分析和影响评估
       │
       ▼
  [24小时内] 完成修复代码编写和审查
       │
       ▼
  [48小时内] 完成修复验证和部署（重新生成所有受影响 HTML）
       │
       ▼

在热修复过程中：
- 如果漏洞涉及数据碎片经济系统（如 0day 无限发现漏洞），在修复部署前**暂停暗网市场的 0day 交易功能**。

- 修复完成后立即在 `CHANGELOG.md` 中记录，并在项目根目录更新 `SECURITY_ADVISORY.md`。

## 5. 响应时间承诺

### 5.1 按严重度分级的 SLA 时间表

| 严重度 | 首次响应 | 修复发布 | 公开披露（自修复发布起） |
|--------|---------|---------|---------------------|
| **Critical** | 24 小时 | 7 个自然日（热修复路径：48 小时） | 14 天 |
| **High** | 48 小时 | 30 个自然日 | 30 天 |
| **Medium** | 5 个工作日 | 90 个自然日 | 90 天 |
| **Low** | 10 个工作日 | 180 个自然日或下一大版本 | 180 天 |
| **Informational** | 15 个工作日 | 无强制修复期限 | 无需披露 |

**术语说明**：

- **首次响应**：安全团队确认收到报告并完成初步审核（Triage 完成）的时间。响应内容包括：确认漏洞真实性、调整（或确认）严重度、分配漏洞编号、预计修复时间。
- **修复发布**：修复代码已合并并部署的时间。对于本项目（纯前端），这意味着修复后的 HTML 文件已更新到发布目录。
- **公开披露**：安全公告公开发布的时间。在披露前，漏洞细节仅安全团队和报告人知晓。披露内容包括：漏洞描述、受影响版本、修复版本、致谢列表、CVE 编号（如适用）。

### 5.2 公开披露策略

#### 5.2.1 CVE 申请流程

对于符合 CVE 申请条件的漏洞（Critical / High，且属于代码实现层面的安全缺陷而非纯架构/设计问题），安全团队将通过 GitHub Security Advisory 的 CVE 申请功能自动提交申请。流程如下：

1. **评估**：在 Confirmed 状态时，安全团队评估漏洞是否有必要申请 CVE。以下情况的漏洞通常不申请 CVE：
   - 纯前端项目中由 `localStorage` 共享架构决定的"设计如此"问题
   - 仅影响开发/测试环境的漏洞
   - 已在内部代码审查中发现且从未发布到公开版本的漏洞
2. **申请**：通过 GitHub Advisory 提交 CVE 申请，MITRE 通常在 1-3 个工作日内分配编号。
3. **预留**：在修复发布前，安全公告中使用 "CVE-YYYY-NNNNN (reserved)" 表示 CVE 已预留。
4. **发布**：修复发布后，更新 CVE 记录为已发布状态，包含完整的漏洞描述和修复信息。

#### 5.2.2 披露时间线协调

安全团队遵循"协调披露"（Coordinated Disclosure）原则：

1. 在修复发布之前，漏洞细节**仅限安全团队和报告人**知晓。报告人不得在修复前公开漏洞细节。
2. 修复发布后，安全团队与报告人协商确定公开披露日期。默认按 SLA 表中的披露时间执行，但可根据报告人的需求适当提前或延后（需双方同意）。
3. 如果报告人希望在学术会议、安全会议（如 Black Hat、DEF CON）上发表相关研究，安全团队将尽力配合调整披露时间线，前提是不影响用户的安全。
4. 如果漏洞在被报告之前已经被公开利用（0day in the wild），安全团队将**立即发布紧急公告**，而非等待常规披露周期。

#### 5.2.3 公告内容

安全公告（Security Advisory）包含以下内容：

```
标题：ISV-YYYY-NNN — [严重度] — [漏洞简述]

发布日期：YYYY-MM-DD
CVE 编号：CVE-YYYY-NNNNN（如适用）

受影响版本：投资模拟游戏系统 V1.0 — V1.x
修复版本：V1.y+

漏洞描述：
[技术性描述漏洞的本质和影响]

致谢：
感谢 [报告人姓名/昵称] 发现并报告此漏洞。

修复说明：
[修复内容的技术摘要]

升级建议：
[用户应采取的行动]
```

### 5.3 延期的豁免情形与沟通机制

在某些情况下，修复发布可能超出 SLA 承诺的时间。以下情形可申请延期，但必须与报告人沟通：

| 豁免情形 | 延期上限 | 适用条件 |
|---------|---------|---------|
| **生成器批量重新生成** | 额外 7 天 | 修复涉及 Python 生成器脚本修改，需重新运行所有生成器并验证 34+ 个 HTML 子页面的一致性和正确性 |
| **多标签页回归测试** | 额外 5 天 | 修复涉及 `localStorage` 同步逻辑，需对所有子页面的跨标签页交互做完整回归测试 |
| **上游依赖等待** | 无固定上限 | 修复依赖外部组件（如浏览器 Bug 修复、第三方库更新） |
| **不可抗力** | 协商确定 | 核心开发者不可用（疾病/假期）、自然灾害等 |
| **低严重度降级** | 按降级后 SLA | 在 Triage 阶段重新评估后严重度降级，按降级后的 SLA 执行 |

**沟通机制**：

1. 一旦确定需要延期，安全团队必须在原 SLA 到期前至少 3 个工作日通知报告人。
2. 通知内容包括：延期原因、新的预计完成时间、当前进展。
3. 如果报告人对延期有异议，可在 5 个工作日内提出。安全团队将在收到异议后 3 个工作日内回复调整方案。
4. 连续两次延期后仍未完成修复的，安全团队应主动提议将漏洞的公开披露日期提前（让社区知晓风险），或请求社区协助修复。

## 6. 赏金与致谢计划

### 6.1 漏洞赏金等级表

本项目设立漏洞赏金计划，对符合条件的漏洞报告人提供现金奖励。赏金金额根据漏洞的严重度和实际影响确定：

| 严重度 | 赏金范围（人民币） | 典型漏洞示例 |
|--------|:---------:|------|
| **Critical** | ¥5,000 — ¥20,000 | 母锁令牌伪造、全局 XSS、三层锁链全局绕过、数据碎片经济系统破坏 |
| **High** | ¥1,500 — ¥5,000 | 单模块 XSS、P2P 违约率篡改、子锁/数据锁绕过、FBI 突袭机制绕过 |
| **Medium** | ¥500 — ¥1,500 | CSS 注入/UI 欺骗、交易日志信息泄露、竞态条件、HTML 结构安全缺陷 |
| **Low** | ¥100 — ¥500 | localStorage 容量耗尽、调试信息残留、不安全的 Math.random() |
| **Informational** | 无现金奖励 | 安全建议、最佳实践推荐、代码质量改进建议 |

**赏金调整因素**：

- **报告质量**：提供完整 PoC + 修复建议的报告可获得上限赏金；仅描述问题但无复现步骤的报告获下限赏金。
- **漏洞影响面**：影响全部模块的漏洞获上限；仅影响单个子页面的漏洞获下限。
- **攻击利用难度**：无需任何前置条件即可利用的漏洞获上限；需要复杂攻击链的漏洞获下限。
- **首次报告**：同一漏洞的首次报告人获得全额赏金；后续重复报告不计赏金但可获致谢。

**赏金发放流程**：

1. 漏洞在 Closed 状态后，安全团队在 15 个工作日内确定赏金金额并通知报告人。
2. 报告人提供收款信息（支付宝/银行转账/PayPal）。
3. 财务审批（项目预算范围内）后，在 10 个工作日内完成转账。
4. 赏金发放完成后，在安全公告中注明（如报告人选择匿名则不注明）。

**不适用赏金的情形**：

- 内部开发团队成员（含承包商）报告的漏洞——走内部 Bug Bounty 流程。
- 通过自动化扫描工具批量提交且未经验证的漏洞。
- 纯理论性漏洞（无可行的复现步骤）。
- 已公开的已知漏洞（如在公开 issue 中已讨论的安全问题）。
- 拒绝服务（DoS）类漏洞——因为纯前端应用无服务端，无法被"拒绝服务"。
- 需要物理访问用户设备才能利用的漏洞。

### 6.3 Safe Harbor 条款（善意测试者的法律保护承诺）

本项目承诺，对于遵循本策略且出于善意目的进行安全测试的研究者，项目维护方将：

1. **不提起法律诉讼**：对于善意发现并报告漏洞的安全研究者，项目方不会因其安全测试行为提起任何法律诉讼或向执法机构举报。

2. **视为授权测试**：在本策略条款范围内进行的安全测试将被视为经过授权。测试范围限定为：
   - 按照漏洞报告模板中的复现步骤进行最小化验证。
   - 不超出验证漏洞所需的数据访问范围（不对 `localStorage` 数据进行超出验证需要的批量导出）。
   - 不破坏其他用户的游戏数据（由于纯前端项目的单用户特性，此条天然满足）。

3. **数据保护**：如果测试过程中意外接触到了非模拟数据（如真实个人信息），研究者应立即停止测试并安全删除相关数据，同时通知安全团队。项目方承诺此类数据仅限于开发者模拟生成的测试数据。

4. **公开披露保护**：研究者在遵守协调披露原则（在修复发布前不公开细节）的前提下，项目方不会对研究者的公开披露行为进行阻挠或威胁。

5. **免责声明**：本 Safe Harbor 条款不保护以下行为：
   - 超出验证需求的大规模数据泄露或破坏。
   - 利用漏洞获取不当利益（如将漏洞出售给第三方而非报告给项目方）。
   - 在修复发布前公开漏洞细节（违反协调披露原则）。
   - 社会工程学攻击或物理入侵。
   - 对托管本项目代码的第三方平台（如 GitHub）的攻击。

### 7.2 输入验证与输出编码

#### 7.2.1 已有实践巩固

项目中 `escapeHtml()` 函数已在 22+ 个文件中实现，这是防御 XSS 的第一道防线。当前实现：

```javascript
function escapeHtml(str) {
    var d = document.createElement('div');
    d.textContent = str || '';
    return d.innerHTML;
}
```

**优势**：利用浏览器原生 DOM API 的文本节点设置进行转义，比正则表达式更可靠（不会被新型 XSS payload 绕过）。

**当前不足**：

1. **代码重复**：同一函数在 22+ 个文件中各自独立定义，任何一个文件的 `escapeHtml` 如果被遗漏更新，将产生安全缺口。
2. **使用不统一**：部分渲染路径使用 `textContent`（如 `darknet-ws-trade.js` 的标题设置），部分路径使用 `escapeHtml() + innerHTML`（如银行页面的交易流水），缺乏统一规范。
3. **未覆盖所有输出点**：`invest-sim.html` 中有大量 `html += '...' + someValue + '...'` 模式，如果 `someValue` 来自 `localStorage` 且未被 `escapeHtml()` 包裹，将成为注入点。

#### 7.2.2 改进建议

1. **抽取公共模块**：将 `escapeHtml()` 定义为独立的 `shared-security.js` 文件，所有子页面通过 `<script src="shared-security.js">` 引用。

2. **建立安全渲染规范**：
   - 所有来自 `localStorage` 的数据在拼接到 `innerHTML` 前**必须**经过 `escapeHtml()` 处理。
   - 纯文本节点建议使用 `textContent` 直接赋值（无需 `escapeHtml`，性能更优且零注入风险）。
   - 仅在必须渲染富文本（HTML 标签）时才使用 `innerHTML`，且此时需使用 DOMPurify 类库或自定义白名单过滤。

3. **代码审查检查点**：在 PR 审查中对所有 `innerHTML` 赋值做专项检查，确保数据源已安全处理。

### 7.3 凭据管理

#### 7.3.1 当前状态

项目为纯前端游戏，不存在真实的服务端凭据。但以下数据具有"模拟凭据"性质，需要注意：

| 数据类型 | 位置 | 风险 |
|---------|------|------|
| VPN 凭据 | `gen_banks.py` 中的 `vpnCredentials` 字典 | 硬编码在源码中，如果这些凭据曾用于任何真实系统测试则构成泄露 |
| 母锁令牌 `seal` | `invest-sim.html` 的 `refreshMasterToken()` | 使用 `Math.random()` 生成，非密码学安全，但适用于无服务端的游戏场景 |
| 银行卡号 | `gen_banks.py` 和 `gen_accounts_excel.py` | 模拟数据，但卡号格式与真实卡号相似（如 19 位借记卡号） |

#### 7.3.2 改进建议

1. **明确标注模拟性质**：在所有生成器脚本的头部添加注释，明确声明数据仅供游戏模拟使用。

```python
# ⚠️ 安全声明 ⚠️
# 本文件中的所有数据（卡号、姓名、凭据等）均为虚构模拟数据，
# 仅供游戏内使用。这些凭据不可用于任何真实系统的认证。
# 如果本生成器被复用于其他项目，必须替换所有数据为项目自有数据。
```

2. **环境变量替代硬编码**：如果未来项目需要真实凭据（如接入第三方 API），必须使用环境变量而非硬编码：

```javascript
// ✅ 推荐
const API_KEY = process.env.INVEST_SIM_API_KEY;

// ❌ 禁止
const API_KEY = 'sk-1234567890abcdef';
```

#### 7.5.2 跨文件 Schema 一致性校验（本项目特有）

这是本项目最重要的安全测试之一——鉴于 Bug #3 的教训（token 字段名跨文件不一致），应建立自动化的 Schema 校验脚本：

```python
# tools/check_token_schema.py
"""跨文件 token 结构一致性校验"""
import re
import sys

EXPECTED_FIELDS = {'ts', 'seal', 'status'}

def check_file(filepath):
    with open(filepath, 'r', encoding='utf-8') as f:
        content = f.read()
    # 检查所有 token.xxx 的访问模式
    tokens = re.findall(r'token\.(\w+)', content)
    unknown = [t for t in tokens if t not in EXPECTED_FIELDS]
    return unknown

def main():
    files = ['invest-sim.html'] + glob('gen_*.py')
    errors = []
    for f in files:
        unknowns = check_file(f)
        if unknowns:
            errors.append(f'{f}: 使用了非预期字段 {unknowns}')
    if errors:
        print('Schema 校验失败！')
        for e in errors:
            print(f'  - {e}')
        sys.exit(1)
    print('Schema 校验通过 ✓')

if __name__ == '__main__':
    main()
```

此脚本应在每次 PR 提交时自动运行，防止字段名不一致的问题再次出现。
